`@testing-library/react` 的文档建议尽可能避免直接使用 `container`，而是使用 `screen` 或其他查询方法来访问组件的子元素。

这是因为直接使用 `container` 可能会导致测试变得**脆弱**和**难以维护**。如果您的测试依赖于组件的实现细节，例如组件的 `CSS 类名`或 `DOM 结构`，那么当您更改组件的实现时，您的测试可能会失败，即使组件的功能没有受到影响。

相反，使用 `screen` 或其他查询方法来访问组件的子元素，可以使您的测试更加健壮和可维护。这些方法使得您的测试更加关注组件的功能，而不是实现细节。此外，这些方法还可以使您的测试更加易于阅读和理解，因为它们更直观地表达了您正在测试的内容。

当然，并非所有情况下都可以避免使用 `container`。在某些情况下，您可能需要直接访问组件的 DOM 树，例如测试自定义的渲染逻辑或使用第三方库的组件。在这种情况下，使用 `container` 可能是必要的。但是，如果可能的话，尽可能使用 `screen` 或其他查询方法来访问组件的子元素。
